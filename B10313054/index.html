🤔 功能說明

  • xv6缺少把page swap到disk的的功能，所有的process都在main memory執行。有鑑於此，我們希望置入swap的機制。swap的機制可以一定程度上增加degree of multiprogramming的能力，提高cpu的使用率。但是過度的multiprograming又會導致thrashing（如圖）。





  • 為此，我們使用了近似於LRU（如圖）的page replacement 算法NFU來控制victim page的選擇。使得cpu的運行不僅效率高而且穩定。



  • 進一步對我們的功能進行說明。
    在最底層我們建立swap的功能，當物理內存不足時，拿出部分硬盤空間當SWAP分區（虛擬成內存）使用(如圖)。而NFU（not frequently used）算法可以記錄page的使用頻率，將count最少的page視為victim page，swap功能會把這個page swap到disk，從而讓新進來的process有機會運行，解決內存容量不足的問題。當swap分區中的page又被acquire的時候，會從分區中重新將page swap in。




  • 綜上所述，有swap功能的xv6可以把外部的disk 當成main memory使用，從而增加多工的程度，而NFU是一種對swap的控制。這是我們希望達成的功能。



😇 實作過程(修改哪些檔案[含圖片])

實作流程概述

一、swap框架

  • 首先，我們為xv6置入一個paging framework，此framework具有兩個功能：

    1. 使得xv6能夠把page從main memory取出，並儲存在disk（swap區域）。
    2. 當swap區域中的page再次被acquire的時候，xv6能夠把page從disk（swap區域）取出，並存回main memory

  • 每個process管理自己的page的swap in ，swap out，為了方便實作，我們使用了file system 提供的interface，為每一個process建立一個file作為它的swap區域。

I.file框架：creating/writing/reading/deleting swap file

  • 主要修改檔案：fs.c
  • 為proc struc （proc.h）新增一個參數 swapFile，此參數會用一個pointer指向swap out的目的地file
  • 增加interface：

  1. int createSwapFile(struct proc *p)
    新建一個swap file

  2. int readFromSwapFile(struct proc p, char buffer, uint placeOnFile, uint size)
    從swap file裡讀size bytes

  3. int writeToSwapFile(struct proc p, char buffer, uint placeOnFile, uint size)
    從buffer寫到swapfile裡

  4. Int removeSwapFile(struct proc *p)
    刪除swap file


此外，swap out之後應修改標記：

  • #define PTE_PG 0x200 記錄這個page是不是被swap out
  • 把valid-invalid bit重置，表示此page已不在main memory中(如圖)。






III.swap in

  • 重新把page從file中取回，流程如下：




1.執行中的process需要之前被swap out的page
2.產生一個trap,確認是page fault
3.使用寄存器去disk找這個page
4.分配一塊記憶體
5.從找到的file裡把data map回來pagetable
6.回到user mode，process重新執行剛剛失敗的指令。
7.檢查是否超過MAX_TOTAL_PAGES，如果超過了，其他的page需要被視為victim page swap out 為此swap in 留出空間（如圖）








二、控制swap的執行


面對選擇victim page的問題，我們使用的Page replacement algorithms是NFU，設置一個reference bit來記錄age，藉由timer interrupt來更新資料結構。

三、測試和執行


修改xv6原本就有的viewer，加印出：

2.目前已經被趕走的page數量

removeSwapFile(struct proc* p)
{
	//path of proccess
	char path[DIGITS];

	itoa(p->pid, path+ 6);
 	struct inode *ip, *dp;

	{
		end_op();
		return -1;
	}
 	ilock(dp);
 	  // Cannot unlink "." or "..".
	if(namecmp(name, ".") == 0 || namecmp(name, "..") == 0)
	   goto bad;
 	if((ip = dirlookup(dp, name, &off)) == 0)
		goto bad;
	ilock(ip);
 	if(ip->nlink < 1)
		panic("unlink: nlink < 1");
	if(ip->type == T_DIR && !isdirempty(ip)){
		iunlockput(ip);
		goto bad;
	}
 	memset(&de, 0, sizeof(de));
	if(writei(dp, (char*)&de, off, sizeof(de)) != sizeof(de))
		panic("unlink: writei");
	if(ip->type == T_DIR){
		dp->nlink--;
		iupdate(dp);
	}
	iunlockput(dp);
 	ip->nlink--;
	iupdate(ip);
	iunlockput(ip);
 	end_op();
 	return 0;
 	bad:
		iunlockput(dp);
		end_op();
		return -1;
 }

int
createSwapFile(struct proc* p)
{
 	char path[DIGITS];
	memmove(path,"/.swap", 6);
	itoa(p->pid, path+ 6);
     begin_op();
    struct inode * in = create(path, T_FILE, 0, 0);
	iunlock(in);
 	p->swapFile = filealloc();
	if (p->swapFile == 0)
		panic("no slot for files on /store");
 	p->swapFile->ip = in;
	p->swapFile->type = FD_INODE;
	p->swapFile->off = 0;
	p->swapFile->readable = O_WRONLY;
	p->swapFile->writable = O_RDWR;
    end_op();
     return 0;
}
 //return as sys_write (-1 when error)
int
writeToSwapFile(struct proc * p, char* buffer, uint placeOnFile, uint size)
{
	p->swapFile->off = placeOnFile;
 	return filewrite(p->swapFile, buffer, size);
 }

readFromSwapFile(struct proc * p, char* buffer, uint placeOnFile, uint size)
{
	p->swapFile->off = placeOnFile;

🐻 預期成效、情境說明（怎麼用、誰用）


當 CPU 效能低時，系統會想引入更多的 process 讓 CPU 盡可能地工作。但當存有太多 process 時，大部分的工作將會花費在 page fault 造成的 Page Replacement，致使 CPU 效率下降，最後造成 CPU 的效能越來越低。除了降低 Multiprogramming Degree和利用 Working Set Model “預估 ” ，利用 Page Fault Frequency (Ratio) 控制來防止 Thrashing也是解法之一。

  • 預期能使Xv6的Page Table減少 Page Fault損失，避免Thrashing的發生，降低Access Disk的時長，改善Xv6對Page Fault的處置效率。

  • 作業二已經實作了swap，我們希望在接下來的作業三中，可以用locality的角度加強swap的效能。當發生Page Fault需要前往Disk搜尋缺乏的Page時，透過我們這組設定的特殊Swap功能—Spatial Locality，將預期會使用到的後3個Page順道Swap In，示意圖如下：




  • 修改FIFO算法，其建立在FIFO的基礎上，每次swap的时候不止抓取一个page，也把它附近的page抓进来。分成三部分，最上層的API，FIFO和SWAP。

  • API是用戶引發使用和觀察這一功能的界面。控制層用於管理swap的使用，我們使用FIFO決定4個victim page。當window 滿了的時候，就引發swap，在這一過程中，FIFO算法會把最先進來的4個page置換出去。在最底層我們建立swap的功能，當物理內存不足時，拿出部分硬盤空間當SWAP分區（虛擬成內存）使用，從而解決內存容量不足的情況。當某進程向OS請求內存發現不足時，OS會把內存中暫時不用的數據（這裡設定為最先進來的四個page交換出去，放在SWAP分區中（SWAP OUT）。當某進程又需要這些數據且OS發現還有空閒物理內存時，又會把SWAP分區中的四個page交換回物理內存中（SWAP IN）。

  • 修改Swap的程式碼，設定發生Page Fault後選擇最先進來的4個Victim Page進行Swap Out3.在硬碟搜尋欠缺的Page，又因Spacial Locality的特性，命令Swap In時擷取所需Page以外的的後三個Page4.回傳完成Swap的指令，結束Page Fault的Swap程序。

  • 用戶發現自己的process locality很強，就可以運用這一功能防止崩潰。
